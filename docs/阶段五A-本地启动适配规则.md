# 阶段五A：本地启动适配规则

## 范围

阶段五A不调用AI，只比较“路线启动要求”和“用户当前启动条件”，并生成固定7天验证计划。

## 内部状态

| 内部分数 | 页面状态 | 含义 |
|---|---|---|
| 2 | 当前满足 | 用户条件稳定覆盖路线最低要求 |
| 1 | 可以补足 | 用户区间部分覆盖，或有明确补足方式 |
| 0 | 明显冲突 | 当前条件达不到路线最低要求 |
| null | 待验证 | 路线要求或用户条件缺少可比较信息 |

内部数值只参与程序计算，页面不展示。

## 四项比较规则

1. 首笔变现：同时比较最晚收入期限和最低有意义金额，取更弱的一项。
2. 稳定投入时间：用户时间下限达到要求为“当前满足”；只有上限能覆盖为“可以补足”。
3. 首批用户资源：可直接联系人覆盖要求为“当前满足”；人数不足但已有联系人和补足方式为“可以补足”；0人且无法补足为“明显冲突”。
4. 验证预算：预算覆盖最低要求为“当前满足”；预算不足但愿意花钱或合作为“可以补足”；否则为“明显冲突”。

## 路线总结果

- 任一维度为0：暂缓；
- 没有0但存在待验证：待验证；
- 没有0和待验证但存在1：补足后验证；
- 四项全部为2：优先验证。

最大缺口依次取第一个“明显冲突 → 可以补足 → 待验证”的维度。四项全部满足时，仍提示市场需求需要真实验证。

## 路线差异保护

- 结论状态继续由四项固定规则计算，允许不同路线出现相同状态；
- 结论说明必须包含路线名称、最大缺口和可利用条件；
- 最大缺口必须绑定当前路线及其具体启动要求；
- 补足建议优先引用当前路线的最小验证动作、目标用户和获客渠道；
- 即使旧数据中的三组启动要求相同，页面也必须生成三套路线相关的说明，不能套用同一句固定文案；
- 新AI结果的三组可比较启动要求必须各不相同，否则触发一次自动重试。

### 可计算启动门槛

- “待验证”只用于市场证据，不用于路线启动门槛；
- 启动门槛属于低成本试跑假设，必须提供可计算区间，并在页面语义上继续视为需要真实验证的AI估算；
- 变现周期使用“个月”，金额和预算使用“元”，时间使用“小时”，首批用户使用“人”或“个团队”；
- 周期、金额、时间、预算和用户数必须包含阿拉伯数字；
- 三条路线由上述可比较字段组成的启动要求组合必须各不相同，只有渠道不同不能通过校验；
- 任一字段为“待验证”、缺少数字/单位或数值明显异常时，云函数应拒绝结果并自动重试一次。

## 固定7天计划

固定计划正好包含Day 1—Day 7，每天只有一项最小行动，并包含预计用时、完成证据和降级方案。Day 6固定测试付费信号，避免计划只有学习任务。

阶段五B已经接入第4次AI调用；真实分析优先生成个性化计划，本地固定计划继续作为演示模式和AI失败时的降级方案。
