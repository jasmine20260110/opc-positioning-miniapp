# OPC定位神器 P0 开发计划

> 依据：`prd v1.2` 文件夹内 10 份文档，以及当前代码库实际结构。
> 当前代码基线：微信云开发 QuickStart 模板，业务代码尚未开始。
> 计划假设：1 名开发者全职开发，AI 辅助编码；目标是可供测试者扫码体验的 P0 版本。

## 一、结论先行

PRD 的产品主线清楚，但现有“两天 Demo 计划”建立在“已经有 client/server、Router/Agent/Tools 和五个页面”的假设上；当前代码库并不具备这些基础，实际只有 `miniprogram/ + cloudfunctions/` QuickStart 模板。

因此建议：

1. P0 沿用现有微信原生小程序和云函数，不在此阶段另建 Express 独立后端。
2. 真实 AI 可体验版按 6 个开发日 + 1—2 个缓冲日规划。
3. 若必须两天交付，只做固定演示数据版，暂不承诺真实 AI 稳定性。
4. 首要验证指标不是“报告看起来厉害”，而是用户是否理解依据、确认路线并点击“开始 Day 1”。

## 二、P0 范围

### 必须完成

- 首页和隐私提示；
- 20 题问答、返回修改和本地恢复；
- 3 次串行分析：证据提取、3 条路线、市场机会；
- 每条路线正好 5 项市场机会；
- 4 项启动条件确认；
- 固定规则生成个人启动适配；
- 用户确认路线后生成 7 天行动计划；
- 点击“开始 Day 1”的本地行为记录；
- AI 超时、解析失败和固定演示数据降级；
- 真机体验版验收。

### 明确不做

- 微信登录、手机号授权和账号体系；
- 数据库长期保存与跨设备同步；
- 打卡页、Day 7 自动总结、订阅消息；
- 知识库、五种时间折叠分析和外部报告导入；
- 数据看板、分享海报和复杂动画。

## 三、开发前必须冻结的 5 件事

1. **唯一数据契约**：AI 原始 JSON 用 `snake_case`；小程序内部统一转换为 `camelCase`。
2. **追问冲突**：20 题文档中的“AI 追问触发”不进入 P0；只显示固定补充提示，不增加第 5 次 AI 调用。
3. **技术路线**：P0 以 `miniprogram + cloudfunctions` 为准；架构文档中的 `client/server` 目录只参考分层思想，不照搬目录。
4. **AI 供应商**：在编码前确定 endpoint、Key、两个模型名和 JSON 输出能力；Key 只能存在云函数环境中。
5. **测试账号与环境**：确认真实 AppID、云开发环境 ID、体验成员微信号和可用额度。

成功标志：以上内容写成一页“冻结清单”，开发期间新增需求先放入 P1，不插入 P0。

## 四、推荐技术实现

### 小程序端

```text
miniprogram/
├── pages/
│   ├── index/       首页
│   ├── question/    20题
│   ├── loading/     三次AI处理进度
│   ├── report/      evidence / market / startup / fit 四种状态
│   └── plan/        7天计划
├── services/
│   ├── cloudApi.js  云函数调用唯一入口
│   └── storage.js   本地会话读写
├── utils/
│   ├── questions.js
│   └── demoData.js
└── styles/
```

### 云函数端

```text
cloudfunctions/opcApi/
├── index.js                 入口和 action 分发
├── ai/
│   ├── client.js            AI 请求唯一出口
│   ├── modelRouter.js
│   ├── extractEvidence.js
│   ├── generateRoutes.js
│   ├── analyzeMarket.js
│   └── generatePlan.js
├── prompts/                 4 个 Prompt
├── rules/startupFitRules.js
├── schemas/                 4 种输出校验
└── package.json
```

P0 不强行实现完整 Router → Agent → Tools 框架，但保留三条边界：页面不写 AI 请求、AI Key 不进前端、规则计算不交给大模型。

## 五、6 日主开发计划

### Day 1：冻结契约并搭出可点击页面骨架

任务：

- 整理 Questions、Session、Evidence、Route、StartupFit、Plan 数据结构；
- 新建五个 P0 页面并配置路由；
- 完成首页、统一五步进度、基础颜色和卡片样式；
- 建立本地存储和固定演示数据。

成功标志：不接 AI 也能从首页依次点到计划页，页面不报错。

### Day 2：完成 20 题与本地恢复

任务：

- 按终稿录入 20 题和选项；
- 完成上一题、下一题、提交、进度显示；
- 抽象回答只提示补充，不阻断提交；
- 保存答案、当前题号和当前业务步骤；
- 完成“清除本次记录”。

成功标志：退出小程序再打开，答案和题号仍能恢复；20 题可完整提交。

### Day 3：打通云函数和前两次 AI 调用

任务：

- 新建 `opcApi` 云函数和 AI 请求客户端；
- 配置 Key、模型路由、超时、一次重试和脱敏日志；
- 完成证据提取与 3 条路线生成；
- 增加 JSON 校验和 `snake_case → camelCase` 转换；
- 准备与最终 Schema 一致的演示数据。

成功标志：一组测试答案能稳定得到可解析证据和正好 3 条路线，且每条路线有用户原文依据。

### Day 4：完成市场分析与结果页

任务：

- 完成第 3 次 AI 调用；
- Loading 页显示三个真实处理阶段；
- report 页完成 `evidence` 和 `market` 两种状态；
- 市场卡严格限制 5 个字段；
- 无依据的内容统一展示“待验证”。

成功标志：从 20 题提交到 3 张市场机会卡全链路跑通；失败时可以重试或使用演示数据。

### Day 5：完成启动适配与 7 天计划

任务：

- 完成 4 项启动条件表单和 Q20 时间预填；
- 实现 `startupFitRules`，0/1/2 只在内部使用；
- report 页完成 `startup` 和 `fit` 两种状态；
- 用户查看适配结果后才能确认路线；
- 完成第 4 次 AI 调用和 7 天计划页；
- 记录“开始 Day 1”。

成功标志：同一用户对三条路线得到不同的适配结果；计划恰好 7 天，每天有行动、用时、证据和降级方案。

### Day 6：回归、真机和体验版

任务：

- 使用三组指定场景完整回归；
- 检查返回修改、切后台恢复、AI 超时、JSON 异常；
- 检查 Key、日志和隐私提示；
- 部署云函数、配置环境、上传体验版；
- 用非开发者体验成员微信号跑完整流程。

成功标志：体验成员可扫码完成五步，没有阻断错误；AI 失败时演示仍可继续。

### 缓冲 Day 7—8

只处理阻断体验的问题：真机样式、云函数权限、AI 响应兼容、数据恢复和严重崩溃。不得在缓冲期加入 P1 功能。

## 六、依赖与关键路径

```text
冻结数据契约
→ 20题和本地会话
→ 云函数与AI客户端
→ 证据
→ 3条路线
→ 市场机会与启动要求
→ 固定规则适配
→ 路线确认
→ 7天计划
→ 真机体验版
```

外部依赖：真实 AppID、云环境 ID、AI Key 与模型、体验成员微信号。任何一项未准备好，都可能阻塞真机交付。

## 七、容量估算

假设 1 人每天有效开发 8 小时，6 天共 48 小时；预留约 20% 处理未知问题，可承诺容量约 38 小时。

| 模块 | 预计工时 |
|---|---:|
| 契约与页面骨架 | 6h |
| 20题与本地恢复 | 7h |
| 云函数、AI客户端和4次调用 | 11h |
| 结果页、适配规则和计划页 | 9h |
| 测试、修复与部署 | 5h |
| **合计承诺** | **38h** |

如果实际开发者对微信小程序或云函数不熟，建议直接按 8—10 个工作日排期。

## 八、两天应急版

只有在截止时间硬性要求两天时使用：

- Day 1：五个页面、20 题、本地存储、固定演示 JSON、证据/市场卡展示；
- Day 2：启动条件、规则适配、固定 7 天计划、真机体验版。

应急版可以验证流程和界面，但不能证明真实 AI 输出质量。演示时必须明确“当前为固定测试数据”，不能把它描述成已完成的真实智能分析。

## 九、主要风险与处理

| 风险 | 等级 | 处理 |
|---|---|---|
| 文档架构与代码仓库不一致 | 高 | 以现有云开发代码为 P0 技术基线，回写技术文档 |
| AI 输出不是合法 JSON | 高 | Schema 校验、一次重试、固定演示数据降级 |
| 市场分析被误当真实调研 | 高 | 强制显示证据来源和“待验证” |
| 20 题填写流失 | 高 | 不设字数门槛、本地保存、先用 3 名用户测时长 |
| 真机才出现权限或环境问题 | 高 | Day 3 就部署第一版云函数，不拖到最后一天 |
| 敏感信息进入日志或代码 | 高 | Key 只进云端配置，日志不记录完整答案 |
| 需求继续膨胀 | 高 | P1 只记录，不进入当前开发任务 |

## 十、P0 完成定义

以下条件必须全部满足：

1. 非开发者体验成员能扫码进入；
2. 不登录也能完成 20 题和后续流程；
3. 恰好生成 3 条路线，判断可追溯到用户原文；
4. 每条路线只展示 5 项市场机会；
5. 启动适配展示状态、理由、最大缺口和补足建议，不显示精确分数；
6. 用户查看适配后才能选择路线；
7. 7 天计划含至少一项市场或付费验证；
8. 点击“开始 Day 1”有明确反馈；
9. 返回、退出再进入后，关键数据不丢失；
10. AI 失败时可以重试或进入明确标注的演示模式。

## 十一、建议立即执行的第一步

先不要写页面。先用半天完成“数据契约冻结 + 技术路线确认 + 外部配置核对”，再以一组固定测试答案跑通完整 JSON 数据流。数据流稳定后再接 UI，可以避免前后端反复返工。
