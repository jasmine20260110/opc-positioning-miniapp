# OPC定位神器 MVP Demo 阶段开发计划

> 目标：只做内部演示，简单实现 P0 主链路，不按外部正式产品标准建设。

## 一、总体策略

- 沿用当前微信原生小程序：`miniprogram + cloudfunctions`。
- 只建一个业务云函数 `opcApi`，不另建 Express 后端。
- 用户数据只保存在小程序本地，不做登录、数据库和跨设备同步。
- AI Key 只放云函数，不能写进小程序代码。
- 先用固定 JSON 跑通完整流程，再接真实 AI；演示时保留固定数据降级开关。
- 只承诺 P0 主流程可走通，不追求正式产品的扩展性和完整测试覆盖率。

预留约 20% 的开发容量，只处理联调、真机差异和阻断演示的问题，不增加新功能。

## 二、MVP 必做范围

1. 完成 20 题问答；
2. 提取心流和优势证据；
3. 生成正好 3 条候选路线；
4. 每条路线展示正好 5 项市场机会；
5. 确认收入、时间、用户资源、预算 4 项启动条件；
6. 用固定规则生成个人启动适配；
7. 用户选择一条路线；
8. 生成 7 天行动计划；
9. 点击“开始 Day 1”后给出明确反馈；
10. AI 失败时可使用固定演示数据继续流程。

## 三、明确不做

- 微信登录、手机号授权和用户账号；
- 云数据库、历史报告和跨设备同步；
- 打卡页、Day 7 总结和消息提醒；
- 知识库、五种时间折叠、外部报告导入；
- 正式埋点、数据看板、分享海报；
- 完整 Router → Agent → Tools 工程框架；
- 外部用户所需的运营后台和正式隐私体系。

## 四、阶段一：冻结数据和演示边界

### 目标

先确定各页面和 AI 之间传递什么数据，避免边做页面边改字段。

### 任务

- 固定 20 题内容和选项；
- 固定 Evidence、Route、MarketOpportunity、StartupConditions、StartupFit、Plan 数据结构；
- 固定 4 次 AI 调用：证据、路线、市场、计划；
- 明确 P0 不调用 AI 追问，抽象回答只显示固定提示；
- 准备 1 组完整输入和对应的固定演示 JSON。

### 成功标志

一组测试答案能用固定 JSON 表达从证据到 7 天计划的全部结果，字段不再随意改名。

## 五、阶段二：搭建静态完整流程

### 目标

暂时不接 AI，先确认整个 Demo 能从首页走到最后。

### 任务

- 将首页改成 OPC 定位神器入口；
- 新建 question、loading、report、plan 页面；
- report 页用 `evidence / market / startup / fit` 四种状态复用；
- 接入固定演示 JSON；
- 完成页面之间的前进、返回和路线选择。

### 成功标志

在微信开发者工具中，可以用固定数据完整走通：

```text
首页 → 问答 → 证据 → 市场机会 → 启动条件 → 启动适配 → 选择路线 → 7天计划
```

## 六、阶段三：完成 20 题和本地状态

### 目标

让用户真实填写 20 题，并且中途返回不会丢失答案。

### 任务

- 录入最终 20 题、题型和选项；
- 完成上一题、下一题、提交和进度显示；
- 开放题不设置最低字数；
- 回答过于抽象时显示固定建议，但不阻断；
- 使用 `wx.setStorageSync` 保存答案、当前题号和当前步骤；
- Q20 转换成每周可投入时间，供启动条件预填。

### 成功标志

20 题可以完整提交；退出再打开后，题号和已填答案仍能恢复。

## 七、阶段四：接入前三次 AI 分析

### 目标

用真实回答生成证据、3 条路线和市场机会。

### 任务

- 新建 `cloudfunctions/opcApi`；
- 建立 AI 请求唯一入口，配置模型、超时和一次重试；
- 完成证据提取、路线生成、市场分析 3 个 action；
- 校验 AI 返回的 JSON；
- 将 AI 的 `snake_case` 字段转换成小程序使用的 `camelCase`；
- Loading 页展示三个分析阶段；
- 失败时显示“重试”或“使用演示数据”。

### 成功标志

至少一组真实回答能够生成：

- 可引用用户原文的证据；
- 正好 3 条不同路线；
- 每条路线正好 5 项市场机会；
- 没有依据的结论显示“待验证”。

## 八、阶段五：完成启动适配和 7 天计划

### 目标

完成从“分析结果”到“选择方向并开始行动”的闭环。

### 任务

- 完成 4 项启动条件表单；
- 新增 `startupFitRules` 固定规则；
- 0/1/2 只在程序内部使用，页面只显示状态和理由；
- 展示每条路线的最大缺口和补足建议；
- 用户查看适配结果后才能选择路线；
- 完成第 4 次 AI 调用，生成 7 天计划；
- 点击“开始 Day 1”后记录本地状态并提示成功。

### 成功标志

- 三条路线能得到不同的启动适配结果；
- 计划恰好包含 Day 1—Day 7；
- 每天只有一项最小行动，并包含用时、完成证据和降级方案；
- 单日预计用时不超过60分钟，7天预计用时上限之和不超过用户每周可投入时间上限；
- 至少包含一次真实用户或市场验证，并至少包含一次付费信号验证。

## 九、阶段六：联调和 Demo 验收

### 目标

保证演示过程中没有阻断错误。

### 任务

- 使用 3 组测试答案跑完整流程；
- 测试返回修改、退出恢复、AI 超时和 JSON 格式错误；
- 检查 API Key 未进入小程序代码；
- 检查页面没有旧版星级评分、精确适配分数和打卡入口；
- 在微信开发者工具和至少一台真机上完成全流程；
- 准备一键切换固定演示数据的方式。

### 成功标志

连续完成 3 次全流程演示，无崩溃、无空白页；即使 AI 接口失败，也能切换固定数据完成演示。

## 十、阶段依赖

```text
数据结构冻结
→ 静态完整流程
→ 20题与本地状态
→ 前三次AI分析
→ 启动适配与计划
→ 联调验收
```

不能跳过“静态完整流程”直接接 AI，否则 AI 字段变化会同时拖累页面和云函数调试。

## 十一、主要风险

| 风险 | 处理方式 |
|---|---|
| AI 返回格式不稳定 | JSON 校验、一次重试、固定演示数据兜底 |
| 当前仓库只是 QuickStart 模板 | 按 MVP 新建页面和云函数，不套用文档中的完整独立后端架构 |
| 功能继续膨胀 | 不属于十项必做范围的内容全部放入 P1 |
| 20 题填写过程出错或丢失 | 每题保存到本地，阶段三单独验收 |
| 演示现场网络不稳定 | 保留固定演示模式，提前跑通三组数据 |

## 十二、最终完成定义

Demo 完成必须同时满足：

1. 能完整提交 20 题；
2. 能展示带原文依据的证据；
3. 能展示 3 条路线和每条 5 项市场机会；
4. 能确认 4 项启动条件并看到适配结果；
5. 能选择路线并生成 7 天计划；
6. 点击“开始 Day 1”有明确反馈；
7. 返回或退出后关键数据不丢失；
8. AI 失败时仍能完成演示。

## 十三、MVP跑通后待优化项

以下内容不作为 MVP Demo 的完成条件。只有在主流程已经稳定跑通、真实用户验证方向有效后，再按实际需求逐步加入，避免提前增加开发复杂度。

### 1. 完善数据契约

- 统一 Evidence、Route、MarketOpportunity、StartupFit、Plan 等数据结构；
- 明确字段类型、必填项、默认值和异常状态；
- AI 返回的 `snake_case` 统一转换为小程序使用的 `camelCase`；
- 前端、云函数和固定演示数据共用同一套字段定义，减少联调时反复改字段。

### 2. 完善日志、错误处理和 AI 降级

- 记录每次 AI 调用的阶段、耗时和失败原因，但不记录用户敏感信息；
- 区分网络失败、超时、返回格式错误和内容缺失；
- 增加明确的重试提示和固定演示数据降级入口；
- 保证单次 AI 调用失败时，不会导致整个流程空白或崩溃。

### 3. 增加多场景测试数据和详细验收清单

至少保留以下 3 组测试场景：

1. 有心流和能力证据，但市场证据较弱；
2. 市场机会较好，但时间、预算或首批用户资源不足；
3. 多方向能力较强，但收入期限和资源约束明显。

每组都要检查：

- 是否正好生成 3 条路线；
- 心流和优势判断是否引用用户原文；
- 每条路线是否正好包含 5 项市场机会；
- 方向要求和个人启动条件是否区分清楚；
- 启动适配理由是否与固定规则一致；
- 7 天计划是否包含市场验证或付费验证；
- AI 失败时是否仍能完成演示。

### 4. 完善微信体验版交付

- 使用正式小程序 AppID 上传代码；
- 配置云函数、合法域名和必要的环境变量；
- 添加评委或测试人员为体验成员；
- 生成体验版二维码；
- 使用非开发者账号在真机上完成一次全流程验收。

### 5. 逐步优化代码结构

- 外部可以继续保留一个 `opcApi` 云函数，内部按证据、路线、市场、计划和规则拆分模块；
- 统一 AI 请求入口、模型配置和 JSON 校验；
- 避免把页面跳转、业务规则、AI Prompt 和数据存储全部写在一个文件里；
- 只有在外部用户、数据量或多人协作明显增加后，再评估是否拆分独立后端以及 Router、Agent、Tools、Infra 等完整架构。
